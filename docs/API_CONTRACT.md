## 1. Response Shape Convention

Backend endpoints follow **two primary patterns**:

| Pattern | Example | Endpoints |
|---------|---------|-----------|
| **Direct payload** | `{"status": "healthy", "version": "2.0.0", ...}` | `/health`, `/status`, `/api/system/info`, `/api/memory/stats`, `/api/personal/status` |
| **Success wrapper** | `{"success": true, "token": "...", "user": {...}}` | `/auth/login`, `/auth/register`, `/api/keys/update`, `/api/memory/{id}` (DELETE) |

Error responses use either:
- `{"error": "<message>"}` — thrown as `JSONResponse`
- `{"detail": "<message>"}` — FastAPI default `HTTPException` serialization

> **Recommendation for RC-2:** Standardize all endpoints to `{"status": "<ok|error>", "data": {...}, "error": "..."}` wrapper.

## 6. Canonical Response Shape Recommendations (RC-2)

For RC-2, all endpoints should converge to:

```json
{
  "status": "ok" | "error",
  "data": { ... },
  "error": null | "error message",
  "meta": {
    "timestamp": "2026-05-31T00:00:00Z",
    "version": "2.0.0"
  }
}
```

This is aspirational — RC-1 maintains backward compatibility with existing response shapes.
